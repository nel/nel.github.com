Un défi en Ruby !

Coder un middleware Rack afin de calculer en temps réel sur les 5 dernières minutes les urls accèdées via rack et le nombre approximatif de visiteurs uniques.

Ce middleware rack doit fonctionner en mode multicomputer (sur le même réseau interne), multiprocess, multithread. Les données exactes à générer sont détaillées dans la section suivante.

Le rack sera intégré à un serveur web et nous testerons les résultats avec une suite de tests d'intégration qui seront publiée. L'exactitude des résultats et la scalabilité de l'ensemble (performance intrinsèque + maitrise du stockage + complexité de l'algorithme en fonction du nombre de requêtes) serviront à déterminer le gagnant.

Le défi en détail

Détails fonctionnels

Support Multiprocess

Le middleware concatenera l'ensemble des données des process/thread rack du réseau utilisant le middleware.
Cette concatenation peut se faire au niveau du stockage, ou tout autre manière laissée à la libre appréciation de l'implémenteur

Cette concatenation se fera par default entre tous les process mais il est possible de positionner un paramètre env['rack.livetraffic_id'] au niveau de chaque requête pour ne concatener que sur les requêtes ayant cette id.

Information aggrégée

Le middleware rack permettra d'aggréger sur une certain période l'information sur les données suivantes:

  • Le nombre de requêtes sur la période
  • Le nombre de rêquetes/s en moyenne sur la période
  • Visiteurs uniques sur la période (l'algorithme tentera de prendre en compte les connexions "nattées").
  • Le nombre d'accès par hostname sur la période
  • Le nombre d'accès par hostname/path sur la période (c'est à dire les url sans les query string)
  • Url des 30 requêtes les plus lentes sur la période

Période de concatenation

Les données récupérées permettront de déterminer les informations demandées sur une période de 5 minutes glissantes aka les "dernières 5 minutes". Le résultat de cette concaténation n'a pas besoin d'être disponible sous forme de résultat dans la mémoire et/ou le stockage. Le client (cf section le client) peut la calculer quand il en a besoin.

L'implémentation devra faire au mieux pour ne pas être trop sensible à un eventuel décallage de temps entre les serveurs d'un même réseau.

Important Il n'est pas demandé de conserver d'historisation des informations, au contraire un des critères de jugement sera la scalabilité au niveau du stockage s'il y en a. Si vous avez besoin de fournir un script de netoyage mettez le dans un script/clean.rb, le script de netoyage doit également scaler convenablement.

Le client : la lecture des données

Le middleware rack se charge de la récupération des données. Ces données sont lus à la demande par un client.

Le client est un simple disponible à script/rack-top et s'executant avec un appel à ./script/rack-top . livetraffic_id est un paramètre optionnel correspondant au paramètre env['rack.livetraffic_id'] qui permet d'avoir plusieurs silos pour gérer les statistiques de plusieurs applications Rack indépendantes.

Les temps seront donnés en ms et les moyennes seront arrondies à 1 décimale.

L'execution de ./script/rack-top renvoie un json répondant au format suivant:

        {
          "requests": 53,
          "rate": 10.5,
          "uniq_visitor": 4,
          "hostnames":  {
                          "mysite.com": 50,
                          "admin.mysite.com": 3
                        },
          "host_paths": {
                          "mysite.com/foo": 40,
                          "mysite.com/bar": 10,
                          "admin.mysite.com/users": 3
                        },
          "slow_requests":  {
                              "mysite.com/foo?params1=slow": 10,
                              "mysite.com/bar": 9,
                              "mysite.com/bar": 8,
                              "mysite.com/bar": 8,
                              "mysite.com/bar": 8,
                              "mysite.com/bar": 7,
                              "mysite.com/bar": 7,
                              "mysite.com/bar": 7,
                              "admin.mysite.com/users?order=created_at": 6,
                              "admin.mysite.com/users?order=confirmed_at": 6
                            }
        }

Voici le détail des clés :

Clé Type Description
requests number Nombre de requêtes sur la période
rate number Requêtes par seconde en moyenne sur la période
uniq_visitor number Nombre de visiteurs uniques sur la période
hostnames Object Liste exhaustives des hostnames en clé et nombre de requêtes sur celui-ci en valeur
host_paths Object Liste exhaustives des hostnames+path en clé et nombre de requêtes sur celui-ci en valeur
slow_requests Object Les 5 requêtes les plus longues l'url complète en clée et le temps en ms en valeur

Les contraintes de packaging

Configuration et fichier config.ru

Le middleware doit être fournis avec un fichier rackup config.ru fonctionnel et être compatible avec la "norme" rack/rackup. Voici un exemple de fichier config.ru

        use Rack::LiveTraffic, {} #puts here your storage/discovery options if needed

        app = proc do |env|
          [ 200, {'Content-Type' => 'text/plain'}, ['OK'] ]
        end

        run app

Stockage, dépendances et versions

Les seuls stockages externes autorisées sont memcached, redis, mongodb et mysql, ils doivent fonctionner avec les paramètres par défault et sans login/mdp.

Si vous souhaitez utiliser un système de queue externe seul faye, rabbitmq et ØMQ sont autorisés

Si vous avez besoin d'un daemon et/ou de workers merci d'interfacer avec foreman dans un procfile. Ces daemons doivent être codé en ruby et toutes leurs dépenses décrites dans un Gemfile. Les gems devront s'installer sans autre dépendances que les librairies de dev des dépendances externes autorisées.

Deux versions de ruby sont autorisées ruby 1.8.7 p249 et 1.9.3 p125, utilisez un .rvmrc pour préciser.

Récapitulatif du livrable

Le livrable est un répertoire ayant au moins la structure suivante :

        . – script/clean    # Script de netoyage des datas au besoin
        | – script/rack-top # Script de lecture des données qui retourne du JSON
        | – config.ru       # Rackfile pour le middleware (see example)
        | – Gemfile         # Gemfile pour les dépendances ruby
        | – Readme.md       # Description les dépendances externes (mongodb...) et des gotchas 
        | – Procfile        # Procfile pour les éventuels daemons ruby
        | – .rvmrc          # Version de ruby si différente de 1.8.7 p249
        | – ...             # Le reste, middleware Rack, ...

L'évaluation des résultats

Comment allons nous déterminer le gagnant

Seul seront prises en compte les réponses qui correspondent à la description du défi et aux contraintes de packaging

Une grille d'évaluation sera mise en place, elle couvrira UNIQUEMENT les points suivants:

  • Qualité intrinsèque du middleware: threadsafe, multiprocess ?
  • Performance globale du middleware
  • Scalabilité du stockage/reseau, est-ce que les performances sont constantes en fonction du traffic, est-ce que le stockage grossit inconsidérément, est-ce que le netoyage si besoin est bloquant, lourd, lent, fragmente le stockage ...
  • Qualité de l'algorithme de calcul des visiteurs uniques
  • Qualité des calculs par rapport au traffic réel
  • Appréciation globale du code et de la solution

Le jury sera composé de 5 personnes faisant parti ou pas de Dimelo, les membres du jury et leurs collègues ne peuvent pas participer au concours.

Les résultats de leur évaluation seront envoyés à chacun des candidats mais seul celle du gagnant sera publiée sur ce site avec son code. Les participants et le gagnant restent propriétaires de leur code et peuvent en disposer à leur convenance.
